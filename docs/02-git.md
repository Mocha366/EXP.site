# Git/GitHub 運用ルール（site プロジェクト）

このドキュメントでは、EXP. Web サイト（`site/` ディレクトリ）の Git / GitHub 運用方針をまとめます。

## ブランチ戦略

基本となるブランチは次の 2 つです。

- `main`: 公開用の安定したコード。**直接コミットしない**ことを推奨します。
- （任意）`develop`: 本番リリース前の統合作業に使いたい場合に作成します。

日々の作業は、`main` から派生させた **トピックブランチ** で行います。

```text
<user>/<topic>
例:
  mitti37/hero-slider-tuning
  mitti37/about-background
```

## 典型的なフロー

1. `main` を最新にする  
   ```bash
   git checkout main
   git pull origin main
   ```
2. 作業用ブランチを切る  
   ```bash
   git checkout -b mitti37/update-activities-section
   ```
3. コードを変更してコミット  
4. GitHub にプッシュして Pull Request を作成  
5. レビュー後、`main` にマージ

## コミットメッセージのルール

厳密なフォーマットはありませんが、以下を目安にします。

- 基本は **日本語** で書く
- 先頭に目的を表すプレフィックスを付けると読みやすい
  - `feat:` 新機能・セクション追加（例: `feat: 活動実績セクションを追加`）
  - `fix:` バグ修正（例: `fix: Hero セクションのレイアウト崩れを修正`）
  - `style:` 見た目・スタイルのみの変更
  - `refactor:` 挙動を変えないリファクタリング
  - `docs:` ドキュメントや README の変更
  - `chore:` 依存関係更新などその他の雑多な変更

例:

```text
feat: 活動の様子に写真ギャラリーを追加
refactor: AchievementsSection を年ごとのアコーディオンに分割
docs: site/docs を EXP. 用に書き換え
```

## PR（Pull Request）の方針

- 1 PR あたりの変更は「1 つの目的」に絞る
- スクリーンショットや GIF を貼るとレビューしやすくなります（特に UI の変更時）
- レビューコメントには「なぜそうしたいか」を簡単に書くと、後から読み返したときにも役立ちます。

## CI/CD

このリポジトリでは、Next.js サイトのデプロイ先として Vercel / GitHub Pages などを利用する想定です。

- `main` ブランチにマージされたものを自動的に本番にデプロイする運用を推奨します。
- 将来的に GitHub Actions を導入する場合は、
  - Lint / Format チェック
  - TypeScript の型チェック
 などを PR のタイミングで実行できると安心です。

# Git/GitHub の運用
このドキュメントでは、TERA プロジェクトにおける Git および GitHub の運用方法について説明します。

## ブランチ戦略
常に存在するブランチは以下の二つです。
- `main`: 本番環境にデプロイされる安定版コードを保持します。**直接コミットしないでください**。
- `develop`: 開発中のコードを統合するブランチです。新しい機能や修正はここにマージされます。

上記二つのブランチに加えて、以下のようなブランチを作成して運用します。
- `<人名>/<内容>`: 作業するためのブランチです。誰が何をするのかが分かる名前にしてください（例: `suzuki3jp/implement-login`）。

## コミットメッセージのルール
コミットメッセージのフォーマットは厳格には決めていません。ただし、以下の点に注意するとより良いです。

- 基本的には日本語で書く
- prefix をつけて目的を明確にする
  - `fix: ` バグ修正
  - `feat: ` 新機能の追加
  - `docs: ` ドキュメントの変更
  - `style: ` フォーマットの変更（コードの動作に影響しないもの）
  - `refactor: ` リファクタリング
  - `chore: ` 細かな変更
- 何を変更したのかを説明するとともに、なぜその変更が必要なのかも簡潔に書く

## CI/CD
GitHub Actions を使用しています。
- PR で常に Lint, Format が実行されます。
- Vercel によってすべてのコミットでプレビューがデプロイされます。
- `main` ブランチにマージされると自動的に本番環境にデプロイされます。
- `develop` ブランチにマージされると自動的に開発環境にデプロイされます。
